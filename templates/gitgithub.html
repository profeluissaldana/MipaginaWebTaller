🐙 Git & GitHub: Control de Versiones

La herramienta estándar en la industria para salvar nuestro progreso y trabajar en equipo.

1. Las 3 Áreas Locales de Git

Antes de que nuestro código viaje a internet (GitHub), pasa por tres estados o zonas en nuestra computadora. Es como preparar un paquete para el correo:

  • Working Directory: Tu carpeta de trabajo. Modificas los archivos (Estado: *Modified*).
  • Staging Area: La mesa de preparación. Eliges qué cambios vas a empaquetar con git add (Estado: *Staged*).
  • Local Repository: La caja cerrada. Guardas la foto de tu proyecto en la base de datos local con git commit (Estado: *Committed*).
Esquema del ciclo de vida de un archivo en Git local.

2. De tu Computadora a la Nube

GitHub funciona como el servidor central en la nube. Una vez que guardaste tus cambios localmente (Commit), usas el comando git push para "empujar" ese paquete hacia tu repositorio remoto.

💡 Dato para la clase: Git es el sistema de control de versiones (el software instalado), mientras que GitHub es la plataforma web donde se alojan esos proyectos.
Conexión entre el repositorio local de tu PC y el servidor remoto de GitHub.

🔑 ¿Cómo identificarse en Git? (Local vs Global)

Antes de hacer cualquier copia de seguridad (Commit), Git necesita obligatoriamente saber quién eres. Para eso se usan estos dos comandos:

git config user.name "Tu Nombre"
git config user.email "tu_email@gmail.com"
🔍 ¡Comprobación de alumnos! Si te sientas en una PC y quieres saber qué identidad está configurada en esa ventana de consola antes de empezar, ejecuta: git config user.name git config user.email

Aquí viene la decisión más importante para nuestro taller en el aula, ya que compartimos computadoras:

⚙️ Configuración Global (--global)

Si agregas el parámetro --global, estás configurando tu identidad para toda la computadora y para todos los proyectos futuros.

⚠️ ¡No recomendado en la escuela! Si otro compañero usa la misma PC después de ti, sus trabajos se guardarán con tu nombre.

💻 Configuración Local (Recomendada para el aula)

Si obvias el --global, la configuración se guarda únicamente dentro de la carpeta de ese proyecto específico.

✅ ¡Ideal para el taller! Cada alumno entra a su proyecto, ejecuta el comando sin "--global" al inicio de la clase, y sus datos quedan grabados solo en su carpeta.

❓ ¿Por qué este método local es el ideal para nuestro taller?
  • 🚀 Evita la fricción administrativa: Como profesor, no tendrás que ir banco por banco entrando al "Administrador de Credenciales de Windows" a borrar cuentas guardadas manualmente al final de cada jornada.
  • 🔑 El comando clave para HTTPS es git config --local credential.helper "": Al dejar el ayudante de credenciales en blanco de forma local, la terminal está forzada a pedir credenciales siempre para ese proyecto sin alterar el resto del sistema.
  • 🔒 Fomenta las buenas prácticas: Tus estudiantes aprenden cómo gestionar la seguridad de sus cuentas cuando comparten infraestructura o equipos de desarrollo.
🛑 El problema de las contraseñas guardadas por Windows

Aunque cambies tu nombre local, Windows tiende a recordar el usuario de GitHub de la última persona que trabajó a través del Administrador de Credenciales. Si no prestamos atención, ¡subirás tu tarea al perfil de tu compañero! Para solucionar esto en el aula de forma definitiva y profesional, aprenderemos dos métodos independientes.

🛠️ Las Dos Soluciones para Subir Código sin Dejar Rastro

Opción A: Autenticación por Token (HTTPS)
Paso 1 Generar tu Token en GitHub (Se hace una sola vez)

Entra a tu cuenta de GitHub desde el navegador y sigue esta ruta: Perfil ➡️ Settings ➡️ Developer Settings ➡️ Personal access tokens ➡️ Tokens (classic) ➡️ Generate new token. Tilda la casilla repo.

⚠️ ¡IMPORTANTE! Copia ese código largo (ghp_...) y guárdalo. GitHub no lo volverá a mostrar.
Paso 2 Desactivar el guardado automático en Windows

Abre la terminal en tu proyecto y ejecuta este comando para borrar el ayudante de Windows en esta carpeta:

git config --local credential.helper ""
Paso 3 Hacer el Push e ingresar las credenciales

Cuando ejecutes git push origin main, la consola te pedirá:

  • Username: Tu usuario de GitHub.
  • Password: Pega aquí el Token (ghp_...) (No tu contraseña web).
🛡️ Opción B: Autenticación por Llaves SSH (La más Profesional)

Con este método, en vez de escribir tokens, creamos un archivo de "llave digital" en tu carpeta. Git la usará para identificarte automáticamente y Windows jamás guardará tu contraseña.

Paso 1 Crear las llaves en la consola

Dentro de la terminal de tu proyecto, escribe lo siguiente (reemplazando tu correo):

ssh-keygen -t ed25519 -C "tu_email@gmail.com"

Presiona Enter a todas las preguntas que te haga la consola para aceptar las opciones predeterminadas.

Paso 2 Vincular tu llave con tu cuenta de GitHub

Copia el contenido de tu llave pública al portapapeles ejecutando este comando:

clip < ~/.ssh/id_ed25519.pub

Ahora ve a tu **GitHub web ➡️ Settings ➡️ SSH and GPG keys ➡️ New SSH key**, ponle un título (ej: "PC Escuela") y pega el código.

Paso 3 Configurar tu proyecto para usar SSH

Por defecto las carpetas se vinculan con HTTPS. Cambia el enlace remoto a SSH usando este comando:

git remote set-url origin git@github.com:TU_USUARIO/TU_REPOSITORIO.git
🎉 ¡Resultados blindados! Usando cualquiera de los dos métodos anteriores, los proyectos quedan aislados. El alumno que se siente en la próxima hora no tendrá acceso a las cuentas de los demás.

👥 Git en Equipo y Buenas Prácticas Profesionales

En proyectos reales no trabajamos solos. Git fue creado justamente para que varias personas puedan colaborar sobre el mismo proyecto sin perder cambios ni sobrescribir el trabajo de otros compañeros.

🚀 Idea clave: Antes de empezar a programar en clase, siempre conviene sincronizar la carpeta local con GitHub usando: git pull origin main
1️⃣ Para trabajar en equipo (Crucial en proyectos grupales)
📥 git pull (Traer y fusionar cambios)

Descarga los últimos cambios desde GitHub y los fusiona automáticamente en la carpeta local del alumno.

⚠️ ¿Por qué es importante?
Si el "Alumno 1" sube cambios a GitHub y el "Alumno 2" sigue trabajando sin actualizarse, Git bloqueará el próximo git push porque la carpeta local estará desactualizada.
git pull origin main
📦 git clone (Clonar un proyecto)

Descarga una copia completa de un repositorio existente de GitHub hacia una carpeta nueva en la computadora.

✅ ¿Cuándo se usa?
Si el proyecto ya existe en GitHub, NO se debe usar git init. El alumno simplemente debe clonar el proyecto.
git clone git@github.com:usuario/proyecto.git
2️⃣ Para no romper la rama principal (Buenas prácticas)

En proyectos profesionales normalmente no se trabaja directamente sobre la rama principal main o master.

Para agregar nuevas funciones sin arriesgar el código que ya funciona, se utilizan ramas (branches).

💡 Ejemplo:
Un alumno puede crear una rama para desarrollar el formulario de login mientras otro compañero trabaja en el diseño visual del sitio.
🌿 git branch y git checkout

Permiten crear y cambiar entre ramas de trabajo.

Crear una nueva rama:
git branch nueva-funcion
Moverse a esa rama:
git checkout nueva-funcion
Forma moderna (crear y cambiar al mismo tiempo):
git checkout -b nueva-funcion
3️⃣ El salvavidas para corregir errores

A todos les pasa: modifican archivos, algo deja de funcionar y no saben cómo volver atrás.

🛟 git restore

Si todavía NO se realizó un git commit, este comando permite descartar cambios recientes y recuperar el último estado seguro del archivo.

⚠️ Importante: Los cambios eliminados con git restore no se pueden recuperar después.
Recuperar un único archivo:
git restore nombre_del_archivo.py
Descartar todos los cambios no guardados de la carpeta:
git restore .
4️⃣ Vital para Flask, Render y despliegues

Aunque no es un comando de Git puro, está totalmente relacionado con el control de versiones y el despliegue de aplicaciones web.

📄 pip freeze > requirements.txt

Genera un archivo llamado requirements.txt con la lista completa de librerías instaladas en el entorno virtual.

pip freeze > requirements.txt
🚨 ¿Por qué es obligatorio?
Plataformas como Render necesitan este archivo para saber qué librerías instalar automáticamente en el servidor.
✅ Buenas prácticas:
Luego de generar el archivo, también debe subirse a GitHub usando: git add ., git commit y git push.

📋 Guía Rápida de Comandos Esenciales

Los comandos que usaremos en cada clase práctica para subir los desafíos del taller:

⭐ Recomendación de Oro del Prof. Luis Saldaña: Al iniciar cada clase, antes de escribir una sola línea de código, abran la terminal y ejecuten un git pull origin main para asegurarse de que su PC se actualice con lo último que subieron la clase anterior (ya sea desde su casa o GitHub web).
Comando Git ¿Para qué sirve en nuestro taller?
git init Inicializa un repositorio local en la carpeta del proyecto. Se hace una sola vez por proyecto. Crea una carpeta .git, el punto hace que esta carpeta esté oculta, pero si voy al administrador de archivo y activo mostrar elementos ocultos, .git aparecerá con un color amarillo claro, que significará que estuvo oculto, luego es recomendable activar nuevamente esta opción de elementos ocultos.
git config user.name Asegurarse de qué usuario de Git está configurado en esa carpeta específica para firmar los commits (los cambios)
git config user.email Asegurarse de qué email de Git está configurado en esa carpeta específica para firmar los commits (los cambios)
git config user.name "Lucho"
git config user.email "lucho@gmail.com"		 Si ejecutan esos comandos sin especificar el nombre/email y les sale el nombre/correo de otro alumno, pueden cambiarlo solo para esa carpeta quitando la palabra --global
git add . Prepara todos los archivos modificados o nuevos (los sube al Staging Area).
git status Nos muestra en qué estado están nuestros archivos locales (si están modificados, listos o guardados). Al mostrarlos y si todavía no se realizó la copia, serán de color verde. Si luego de realizar el git add . agrego un nuevo archivo y ejecuto el git status, este archivo aparecerá en rojo, porque no se ha agregado a la zona de preparación. Una vez realizado el git commit estos archivos dejarán de estar en la zona de preparación, ya no se listarán más porque ya se realizó la copia o backup.
git reset (Quitar archivos del escenario) Si un alumno añade un archivo por error con git add, este comando lo saca de la lista de espera para el commit, sin borrar el trabajo del archivo.
  • Uso típico: git reset nombre_del_archivo.py
  • Por qué importa: Evita que suban basura al repositorio de GitHub.
git diff (Ver cambios línea por línea) Para ver diferencias antes de hacer un Commit.
Muchas veces los alumnos olvidan qué cambiaron exactamente en el código desde el último commit.
Muestra en la consola exactamente qué líneas se borraron (en rojo) y qué líneas se agregaron (en verde) en los archivos modificados.
  • Por qué importa: Les ayuda a revisar su propio código y hacer commits más limpios y descriptivos, en lugar del clásico mensaje "cambios" o "arreglo".
git commit -m "mensaje" Guarda un historial con un comentario explicativo de lo que se hizo (crea una versión fija).
git stash (Guardar en un cajón temporal) Corta los cambios actuales que no se han guardado en un commit y los guarda en una "pila" temporal, dejando el proyecto completamente limpio.
  • Para guardar: git stash (El proyecto vuelve a estar como al principio).
  • Para recuperar los cambios después: git stash pop (Saca los cambios del cajón y los vuelve a poner en los archivos).
  • Por qué importa: El Alumno 1 puede "esconder" su trabajo a medias, dejar que otro grupo use la máquina, y la próxima semana recuperar su progreso idéntico a como lo dejó.
git remote -v Ver a qué repositorio de GitHub está conectada la carpeta de tu proyecto. El -v significa "verbose" (detallado).
git pull origin main (Traer y fusionar cambios) ¡Clave! Descarga y fusiona los cambios más nuevos desde GitHub a tu computadora local. Evita que el sistema se bloquee por falta de sincronización.
git push origin main Sube los cambios locales guardados a la nube en tu cuenta de GitHub.
git log Muestra un historial detallado de los commits realizados con su fecha, hora y el autor.
git restore --source=[codigo_hash] inicio.html Recupera el archivo seleccionado al estado exacto que tenía en ese commit. El codigo_hash se debe colocar sin corchetes y sin comillas, el archivo debe colocarse con su path, preferentemente entre comillas para poder aceptar los espacios en blancos de los nombres de las carpetas en caso que los tuviera.
git clone (Clonar un proyecto) Descarga una copia completa de un repositorio existente en GitHub a una carpeta nueva en la computadora.
  • Por qué es importante: Si un alumno tiene que empezar a trabajar en la computadora del laboratorio y el proyecto ya está en GitHub, no debe hacer git init. Solo debe clonarlo.
  • Uso típico: git clone git@github.com:usuario/proyecto.git
