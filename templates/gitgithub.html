🐙 Git & GitHub: Control de Versiones

La herramienta estándar en la industria para salvar nuestro progreso y trabajar en equipo.

1. Las 3 Áreas Locales de Git

Antes de que nuestro código viaje a internet (GitHub), pasa por tres estados o zonas en nuestra computadora. Es como preparar un paquete para el correo:

  • Working Directory: Tu carpeta de trabajo. Modificas los archivos (Estado: *Modified*).
  • Staging Area: La mesa de preparación. Eliges qué cambios vas a empaquetar con git add (Estado: *Staged*).
  • Local Repository: La caja cerrada. Guardas la foto de tu proyecto en la base de datos local con git commit (Estado: *Committed*).
Esquema del ciclo de vida de un archivo en Git local.

2. De tu Computadora a la Nube

GitHub funciona como el servidor central en la nube. Una vez que guardaste tus cambios localmente (Commit), usas el comando git push para "empujar" ese paquete hacia tu repositorio remoto.

💡 Dato para la clase: Git es el sistema de control de versiones (el software instalado), mientras que GitHub es la plataforma web donde se alojan esos proyectos.
Conexión entre el repositorio local de tu PC y el servidor remoto de GitHub.

🔑 ¿Cómo identificarse en Git? (Local vs Global)

Antes de hacer cualquier copia de seguridad (Commit), Git necesita obligatoriamente saber quién eres. Para eso se usan estos dos comandos:

git config user.name "Tu Nombre"
git config user.email "tu_email@gmail.com"

Aquí viene la decisión más importante para nuestro taller en el aula, ya que compartimos computadoras:

⚙️ Configuración Global (--global)

Si agregas el parámetro --global, estás configurando tu identidad para toda la computadora y para todos los proyectos futuros.

⚠️ ¡No recomendado en la escuela! Si otro compañero usa la misma PC después de ti, sus trabajos se guardarán con tu nombre.

💻 Configuración Local (Recomendada para el aula)

Si obvias el --global, la configuración se guarda únicamente dentro de la carpeta de ese proyecto específico.

✅ ¡Ideal para el taller! Cada alumno entra a su proyecto, ejecuta el comando sin "--global" al inicio de la clase, y sus datos quedan grabados solo en su carpeta.

❓ ¿Por qué este método es el ideal para nuestro taller?
  • 🚀 Evita la fricción administrativa: Como profesor, no tendrás que ir banco por banco entrando al "Administrador de Credenciales de Windows" a borrar cuentas guardadas manualmente al final de cada jornada.
  • 🔑 El comando clave es git config --local credential.helper "": Al dejar el ayudante de credenciales en blanco de forma local, la terminal está forzada a pedir usuario y contraseña (token) siempre para ese proyecto, pero no altera el comportamiento del resto del sistema operativo.
  • 🔒 Fomenta la seguridad informática: Tus estudiantes aprenden el concepto de los PAT (Personal Access Tokens), que es exactamente la forma en la que se trabaja hoy en día en las empresas para conectar servidores y aplicaciones de manera segura.

¡Con este bloque implementado, la guía de Git queda completamente blindada contra cualquier confusión en el laboratorio!

🛑 El problema de las contraseñas guardadas por Windows

Aunque cambies tu nombre local, Windows tiende a recordar el usuario de GitHub de la última persona que trabajó. Si no prestamos atención, ¡subirás tu tarea al perfil de tu compañero!. Para evitar tener que borrar las credenciales de Windows en cada clase, les enseñaré a autenticarse usando Tokens de Acceso Personal (PAT) o configurando la autenticación por consola para cada commit, asegurando que cada uno suba el código a su propia cuenta de GitHub sin conflictos.

🚀 La Solución: Autenticación por Token en cada clase

Para evitar conflictos, utilizaremos un Token de Acceso Personal (PAT) de GitHub. Este token es una clave larga que reemplaza a tu contraseña tradicional y que solo tú debes conocer.

🛠️ Paso a Paso: Cómo subir tu código sin dejar rastro en la PC

Paso 1 Generar tu Token en GitHub (Se hace una sola vez)

Entra a tu cuenta de GitHub desde el navegador y sigue esta ruta:

  1. Ve a tu foto de perfil (arriba a la derecha) ➡️ Settings (Configuración).
  2. Baja en el menú izquierdo hasta el final y haz clic en Developer Settings.
  3. Selecciona Personal access tokens ➡️ Tokens (classic).
  4. Haz clic en Generate new token (classic).
  5. Ponle un nombre (ej: "ClaseComputacion") and tilda la casilla principal que dice repo (esto te da permisos para subir código).
  6. Haz clic abajo de todo en Generate token.
⚠️ ¡IMPORTANTE! Copia ese código largo que aparece (empieza con ghp_...) y guárdalo en un bloc de notas o envíatelo por correo. GitHub NO te lo volverá a mostrar nunca más.
Paso 2 Desactivar el guardado automático en Windows

Antes de hacer el git push, abre la terminal en tu proyecto y ejecuta este comando comando para decirle a Git: "No guardes ninguna contraseña en esta computadora":

git config --local credential.helper ""

Esto asegura que cuando termine la clase, nadie más pueda usar tus accesos.

Paso 3 Hacer el Push e ingresar las credenciales

Cuando ejecutes el comando para subir tu código por primera vez en la clase:

git push origin main

Git abrirá una ventana flotante o te pedirá en la consola dos datos:

  • Username: Escribe tu usuario de GitHub.
  • Password: NO pongas tu contraseña habitual. Pega aquí el Token (ghp_...) que generaste en el Paso 1.
🎉 ¡Listo! Tu código se subirá de forma segura a tu propia cuenta. En la siguiente clase, el próximo alumno que use la PC tendrá que poner sus propios datos porque gracias al Paso 2, la máquina no recordó los tuyos.

📋 Guía Rápida de Comandos Esenciales

Los comandos que usaremos en cada clase práctica para subir los desafíos del taller:

⭐ Recomendación de Oro del Prof. Luis Saldaña: Al iniciar cada clase, antes de escribir una sola línea de código, abran la terminal y ejecuten un git pull origin main para asegurarse de que su PC se actualice con lo último que subieron la clase anterior (ya sea desde su casa o GitHub web).
Comando Git ¿Para qué sirve en nuestro taller?
git init Inicializa un repositorio local en la carpeta del proyecto. Se hace una sola vez por proyecto.
git add . Prepara todos los archivos modificados o nuevos (los sube al Staging Area).
git commit -m "mensaje" Guarda un historial con un comentario explicativo de lo que se hizo (crea una versión fija).
git pull origin main ¡Clave! Descarga y fusiona los cambios más nuevos desde GitHub a tu computadora local. Evita que el sistema se bloquee por falta de sincronización.
git push origin main Sube los cambios locales guardados a la nube en tu cuenta de GitHub.
git status Nos muestra en qué estado están nuestros archivos locales (si están modificados, listos o guardados).
git log Muestra un historial detallado de los commits realizados con su fecha, hora y el autor. De aquí copiamos el código hash (ID del commit) que necesitamos para restaurar archivos.
git restore --source=[codigo_hash] inicio.html Recupera el archivo seleccionado (en este caso inicio.html) al estado exacto que tenía en ese commit. Puedes usar un punto (.) en vez del nombre del archivo para restaurar todo el proyecto junto.
